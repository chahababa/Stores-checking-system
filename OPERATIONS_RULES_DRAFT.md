# 營運規則草案

## 1. 目的

這份文件用來整理門市巡檢系統目前與下一階段要採用的營運判讀規則。

重點不是技術實作，而是先把營運語言與判讀邏輯定清楚，讓後續的：

- 首頁 dashboard
- 巡店明細
- 歷史列表
- 月報與報表
- 標籤系統
- 通知機制
- 客訴串接

都能用同一套規則運作。

---

## 2. A / B / C 評級規則

### 2.1 單題評級

巡店單題原始分數維持 `1 / 2 / 3`，但前台主要顯示改用：

- `3 分 = A`
- `2 分 = B`
- `1 分 = C`

判讀語意：

- `A`：達標
- `B`：需改善
- `C`：明顯異常

### 2.2 分類評級

每個大分類都要有自己的平均分數與等級，不應只顯示整店平均。

分類平均規則：

- `平均 >= 2.5 -> A`
- `平均 >= 1.5 且 < 2.5 -> B`
- `平均 < 1.5 -> C`

分類評級是店長最重要的判讀層，因為它能直接指出：

- 人員管理好不好
- 內場作業穩不穩
- 外場服務有沒有問題
- 餐點品質是不是弱項

### 2.3 整店總評

整店總評也用 `A / B / C` 顯示，但不能只靠整體平均分數直接換算。

建議流程：

1. 先用全題平均算出基礎總評
2. 再套用標籤與風險規則做降級

基礎總評：

- `平均 >= 2.5 -> A`
- `平均 >= 1.5 且 < 2.5 -> B`
- `平均 < 1.5 -> C`

但最終總評還要受標籤規則影響，避免「平均看起來不差，但其實踩到高風險題目」。

---

## 3. 標籤對總評的影響

### 3.1 必查項目

必查項目代表：

- 四間店都通用
- 絕對不能犯錯
- 常見於食安、衛生、法規、基本服務底線

建議規則：

- 任一 `必查項目 = C`：
  - 整店總評最高只能是 `B`
- `必查項目` 有 `2 題以上 = B 或 C`：
  - 整店總評直接降一級

### 3.2 本月加強

本月加強代表：

- 近期特別要求加強追蹤
- 可能來自管理重點、活動檔期、近期品質波動

建議規則：

- `本月加強` 有 `2 題以上 = B 或 C`：
  - 整店總評降一級

### 3.3 客訴項目

客訴項目代表：

- 近期曾被客人投訴
- 對店長的感受最強，因為它直接關係到實際顧客體驗

建議規則：

- 任一 `客訴項目 = C`：
  - 整店總評最高只能是 `B`
- `客訴項目` 有 `2 題以上 = B 或 C`：
  - 整店總評降一級

### 3.4 連續低分

如果同一題持續多週落在低分，應視為比單次低分更嚴重。

建議規則：

- 同一題連續 `2 週` 以上落在 `B / C`：
  - 在 dashboard 顯示「連續低分」
- 同一店 `連續低分題目 >= 3`：
  - 整店總評降一級

### 3.5 降級原則

為避免規則過度複雜，先採以下原則：

- 每次評分最多連續降 `2 級`
- 不會出現比 `C` 更低的等級
- 被降級的原因要能在明細頁與報表頁清楚列出

## 4. 客訴項目自動標記規則

目前系統已經支援手動標記 `客訴項目`。
未來串接客訴資料庫後，建議自動標記規則如下。

### 4.1 時間範圍

預設看最近 `30 天` 的客訴資料。

### 4.2 對應邏輯

若客訴資料能對應到巡店題目，系統應自動建立或更新 `客訴項目` 標籤。

建議欄位：

- 客訴日期
- 客訴店別
- 客訴類型
- 對應題目
- 來源：`complaint_sync`

### 4.3 標記範圍

客訴標記建議優先支援：

- 指定單店
- 指定月份
- 指定題目

如果某個客訴類型屬於全店風險，之後再考慮擴充成全店共用。

### 4.4 自動解除規則

不要讓客訴標記永久殘留。

建議：

- 若超過 `30 天` 沒有新客訴，且未被手動保留
  - 可自動失效
- 若被 owner / manager 手動調整為保留
  - 就不要自動移除

### 4.5 優先級

若同一題同時是：

- 必查項目
- 本月加強
- 客訴項目

前台應同時顯示多個標籤，不能只保留其中一個。

---

## 5. 店長首頁優先指標

店長首頁不應只是資料摘要，而應像「今天要先處理什麼」的工作台。

建議優先順序如下：

### 第一層：今天最需要關心

- 本店總評 `A / B / C`
- 最弱分類
- 待改善任務數
- 必查異常數
- 客訴項目異常數

### 第二層：今天先做什麼

- 今天先看這三件事
- 本店待改善清單
- 最近低分提醒
- 連續低分題目

### 第三層：營運脈絡

- 本月巡店次數
- 最近一次巡店日期
- 最近一次總評
- 本店在職組員數

### 不建議放太前面的資訊

這些可以保留，但不應該是首頁第一眼主角：

- 純原始平均分數
- 太多跨月統計
- 太細碎的明細表

## 6. 通知觸發條件草案

通知機制先以「真正需要被提醒的事」為原則，不要做成大量噪音。

### 6.1 即時通知

建議觸發：

- 新增巡店後出現 `必查項目 = C`
- 新增巡店後出現 `客訴項目 = C`
- 新增巡店後整店總評被降級
- 同題連續第 `2 週` 仍為低分

### 6.2 每日摘要

發給店長 / 主管的日摘要可包含：

- 待改善任務數
- 新增低分題目數
- 必查異常數
- 客訴追蹤數

### 6.3 每週摘要

發給 owner / manager 的週摘要可包含：

- 各店總評
- 各店最弱分類
- 必查異常排行
- 客訴異常排行
- 連續低分題目排行

### 6.4 通知優先級

建議分成三層：

- `高`
  - 必查 C
  - 客訴 C
  - 整店總評降到 C
- `中`
  - 本月加強持續低分
  - 連續 2 週低分
- `低`
  - 一般巡店完成通知
  - 常規週報

---

## 7. 頁面呈現建議

### 7.1 首頁

- 顯示總評 `A / B / C`
- 顯示最弱分類
- 顯示標籤異常數
- 顯示「今天先做什麼」

### 7.2 巡店歷史

- 每筆巡店顯示總評 `A / B / C`
- 可額外顯示低分題數或標籤異常提示

### 7.3 巡店明細

- 單題顯示 `A / B / C`
- 分類顯示平均等級
- 顯示總評調整原因
- 顯示標籤 badge

### 7.4 報表

- 各分類平均與等級
- 各店總評比較
- 必查 / 本月加強 / 客訴項目異常統計
- 低分排行與連續低分排行

---

## 8. 建議實作順序

若照功能價值與風險排序，建議如下：

1. 完成 A / B / C 在更多頁面的統一呈現
2. 強化標籤系統在改善追蹤與報表的語義
3. 完成 QA 測試資料清理
4. 補齊管理面 E2E
5. 實作通知機制
6. 最後串接客訴資料庫

---

## 9. 目前建議先決定的事

若要進下一階段實作，最先要拍板的是這 5 件事：

1. `A / B / C` 是否就是最終固定語言
2. 哪些標籤會造成總評降級
3. 客訴標記預設看幾天
4. 店長首頁第一眼要顯示哪 4 個核心指標
5. 通知是否先做站內，再做 LINE / 外部通知
