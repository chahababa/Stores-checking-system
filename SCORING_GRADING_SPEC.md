# A/B/C 評級規格

## 1. 目標

目前系統主要以 `1 / 2 / 3` 與「全店平均分數」呈現巡店結果。這種顯示方式對店長與主管來說不夠直觀，難以快速判斷：

- 這家店現在整體表現是好還是差
- 哪一大類最弱
- 哪些問題需要優先處理

本規格的目的，是把巡店評分改造成更容易理解的 `A / B / C` 評級體系，讓店長、主管與 owner 都能更快形成「體感」。

---

## 2. 核心原則

### 2.1 單題評級

巡店題目維持原本資料層分數 `1 / 2 / 3`，但顯示層改為：

- `3 分 = A`
- `2 分 = B`
- `1 分 = C`

這代表：

- `A`：符合標準
- `B`：需改善
- `C`：明顯不合格

### 2.2 類別評級

每個分類都要能顯示自己的平均評級，例如：

- 人員管理：`B`
- 內場作業環境：`C`
- 外場作業環境：`A`

目的不是只看全店平均，而是讓使用者知道「哪一塊弱」。

### 2.3 整店總評

整店仍可顯示總評，但不應只顯示平均分數，而應顯示：

- `A / B / C`

必要時可再補一個次要小字：

- `總評 B`
- `平均分數 2.31`

其中 `平均分數` 退居次要，`總評等級` 才是主要資訊。

---

## 3. 計算規則

## 3.1 單題等級

- `score = 3 -> grade = A`
- `score = 2 -> grade = B`
- `score = 1 -> grade = C`

---

## 3.2 類別平均

每個分類的平均分數，以該分類下所有已評分題目計算平均值。

### 類別平均分數對應等級

- `>= 2.5 -> A`
- `>= 1.5 且 < 2.5 -> B`
- `< 1.5 -> C`

### 補充

這個規則保留原本 1~3 的資料基礎，但讓畫面顯示對人更直觀。

---

## 3.3 整店基礎總評

整店基礎總評，先以「各分類平均」再取平均。

### 基礎總評等級

- `>= 2.5 -> A`
- `>= 1.5 且 < 2.5 -> B`
- `< 1.5 -> C`

---

## 3.4 標籤降級規則

整店總評不能只看平均值，還要考慮高風險標籤題目的表現。

### 必查項目

如果有任何 `必查項目 = C`：

- 整店總評最高只能是 `B`

如果有 `2 題以上必查項目 = B 或 C`：

- 整店總評至少降一級

### 客訴項目

如果最近一個月的 `客訴項目` 中有任何一題為 `C`：

- 整店總評至少降一級

如果 `客訴項目` 有 `2 題以上 = B 或 C`：

- 店長首頁應額外顯示「客訴追蹤中」提醒

### 本月加強

如果 `本月加強` 題目有 `2 題以上 = B 或 C`：

- 整店總評至少降一級

### 連續低分

如果有任一題 `連續 2 週以上低分`，首頁應顯示追蹤提醒，但不直接影響總評。

---

## 4. 呈現規則

## 4.1 首頁 Dashboard

### 店長首頁

優先顯示：

- `本店總評：A / B / C`
- `本店最弱分類`
- `必查異常數`
- `客訴追蹤數`
- `本月加強低分數`
- `最近低分提醒`
- `今天先處理這三件事`

不再把「平均分數 2.xx」當成第一層資訊。

### owner / manager 首頁

優先顯示：

- 各店總評
- 本月最低分類排行
- 必查異常店數
- 客訴追蹤店數
- 跨店低分趨勢

---

## 4.2 巡店歷史列表

每筆巡店紀錄建議增加：

- `總評等級`
- 次要資訊才顯示平均分數

例如：

- `總評 A`
- `平均分數 2.73`

---

## 4.3 巡店明細頁

每個題目顯示：

- `A / B / C badge`

每個分類區塊顯示：

- `分類平均等級`
- `分類平均分數（次要）`

例如：

- `內場作業環境：C`
- `平均分數 1.43`

---

## 4.4 報表頁

報表頁要補強成：

- 各分類平均等級
- 各分類平均分數
- 各分類低分題數
- 各分類趨勢

避免只看「整店平均分數」。

---

## 5. 文案與視覺規則

### 顏色建議

- `A`：綠色 / 穩定
- `B`：暖橘 / 注意
- `C`：紅色 / 高風險

### 呈現規則

- 單題：顯示 `A / B / C`
- 分類：顯示 `A / B / C`
- 店總評：顯示 `A / B / C`
- `1 / 2 / 3` 保留在資料層，不作為主要 UI 呈現

---

## 6. 實作拆分

## Phase 1：計算邏輯

新增 helper：

- `getScoreGrade(score) -> A/B/C`
- `getAverageGrade(value) -> A/B/C`
- `buildCategoryGrades(scores)`
- `buildStoreOverallGrade(scores, tagSummary)`

---

## Phase 2：巡店明細與歷史

調整：

- 巡店明細頁顯示單題 `A/B/C`
- 巡店明細頁顯示分類平均等級
- 巡店歷史頁顯示總評等級

---

## Phase 3：首頁 Dashboard

調整：

- leader 首頁加入總評與最弱分類
- owner / manager 首頁加入跨店總評摘要

---

## Phase 4：報表頁

調整：

- 報表以分類平均為核心
- 整店平均改為次要資訊

---

## 7. 驗證清單

### 功能驗證

- 單題分數是否正確顯示 A/B/C
- 分類平均是否正確計算
- 整店總評是否正確計算
- 必查 / 客訴 / 本月加強是否正確影響總評

### UX 驗證

- 店長是否能一眼知道自己現在是 A / B / C
- 店長是否能一眼知道哪個分類最弱
- manager / owner 是否能快速看出哪家店有高風險

### 回歸驗證

- 既有巡店資料是否仍可正常顯示
- 匯出與報表是否未被破壞

---

## 8. 目前建議結論

這次不要直接把所有頁面一次改完。

建議順序是：

1. 先做 `評級計算 helper`
2. 先把 `巡店明細 + 巡店歷史` 改成 A/B/C
3. 再改 `首頁 Dashboard`
4. 最後改 `報表頁`

這樣風險最低，也最容易驗證成效。
