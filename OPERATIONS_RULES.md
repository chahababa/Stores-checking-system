# 營運規則定稿 v1

## 1. 文件目的

這份文件是門市巡檢營運系統的營運判讀基準。

它的用途是統一以下模組的規則：

- 首頁 Dashboard
- 巡店歷史
- 巡店明細
- 改善追蹤
- 報表分析
- 標籤系統
- 未來通知機制
- 未來客訴資料庫串接

之後若有新功能牽涉到評級、標籤、通知、店長工作台，都以這份為準。

---

## 2. 評級語言

### 2.1 單題分數與等級

系統底層仍保留原始分數 `1 / 2 / 3`，但對使用者的主要顯示語言改為：

- `3 分 = A`
- `2 分 = B`
- `1 分 = C`

判讀語意：

- `A`：達標
- `B`：需改善
- `C`：異常

### 2.2 顯示原則

前台主要畫面應以 `A / B / C` 為主，原始分數只作輔助資訊。

適用頁面：

- 巡店歷史
- 巡店明細
- 首頁 Dashboard
- 報表分析

---

## 3. 分類評級規則

每個大分類都要有自己的平均分數與等級，不能只顯示整店平均。

分類平均等級規則：

- `平均 >= 2.5 -> A`
- `平均 >= 1.5 且 < 2.5 -> B`
- `平均 < 1.5 -> C`

系統應至少對以下分類提供分類等級：

- 人員管理
- 內場作業環境
- 外場作業環境
- 廁所環境
- 餐點品質
- 服務品質
- 餐點品質抽查記錄

說明：

- 店長最重要的不是知道「整店平均 2.31」
- 而是知道「哪一類是 A，哪一類是 C」

---

## 4. 整店總評規則

### 4.1 基礎總評

整店總評先用全題平均算出基礎等級：

- `平均 >= 2.5 -> A`
- `平均 >= 1.5 且 < 2.5 -> B`
- `平均 < 1.5 -> C`

### 4.2 最終總評

最終總評不能只看平均，還要受標籤與風險規則影響。

流程固定為：

1. 先算基礎總評
2. 再套用降級規則
3. 輸出最終總評與降級原因

### 4.3 降級上限

- 每次巡店最多降 `2 級`
- 不會出現比 `C` 更低的等級

---

## 5. 標籤與降級規則

目前正式標籤類型固定為三種：

- `必查項目`
- `本月加強`
- `客訴項目`

### 5.1 必查項目

定義：

- 四間店通用
- 屬於底線與高風險項目
- 例如食安、衛生、法規、不可犯錯項目

規則：

- 任一 `必查項目 = C`
  - 整店總評最高只能是 `B`
- `必查項目` 中有 `2 題以上 = B 或 C`
  - 整店總評降 `1 級`

### 5.2 本月加強

定義：

- 當月特別追蹤
- 可指定全部店別或單店

規則：

- `本月加強` 中有 `2 題以上 = B 或 C`
  - 整店總評降 `1 級`

### 5.3 客訴項目

定義：

- 近期與客訴直接相關
- 可指定全部店別或單店

規則：

- 任一 `客訴項目 = C`
  - 整店總評最高只能是 `B`
- `客訴項目` 中有 `2 題以上 = B 或 C`
  - 整店總評降 `1 級`

### 5.4 連續低分

定義：

- 同一題連續多週落在低分

規則：

- 同一題連續 `2 週` 以上為 `B / C`
  - 標記為 `連續低分`
- 同店 `連續低分題目 >= 3`
  - 整店總評降 `1 級`

### 5.5 同題多標籤

若同一題同時符合多種標籤：

- 必查項目
- 本月加強
- 客訴項目

前台必須同時顯示多個標籤，不可只保留其中一個。

---

## 6. 客訴項目自動標記規則

這一段是未來串接客訴資料庫時的正式規則。

### 6.1 時間範圍

預設看最近 `30 天` 的客訴資料。

### 6.2 自動標記條件

若客訴資料可對應到巡店題目，系統應自動建立或更新 `客訴項目` 標籤。

資料來源應標示為：

- `complaint_sync`

### 6.3 標記範圍

第一版先支援：

- 指定店別
- 指定題目
- 最近 30 天有效

### 6.4 自動解除條件

若超過 `30 天` 沒有新的對應客訴，且該標籤不是手動保留，則可自動失效。

若 owner / manager 明確手動保留，則不應自動移除。

---

## 7. 店長首頁指標優先順序

店長首頁應被視為「單店工作台」，不是資料牆。

### 第一層：第一眼必看

- 本店總評 `A / B / C`
- 最弱分類
- 待改善任務數
- 必查異常數
- 客訴項目異常數

### 第二層：今天先做什麼

- 今天先看這三件事
- 本店待改善清單
- 最近低分提醒
- 連續低分題目

### 第三層：營運脈絡

- 本月巡店次數
- 最近一次巡店日期
- 最近一次總評
- 本店在職組員數

### 不應該成為首頁主角的資訊

- 單純的原始平均分數
- 過多跨月數值
- 沒有行動意義的統計

---

## 8. owner / manager 首頁指標優先順序

owner / manager 首頁應是跨店管理總覽。

第一層應優先顯示：

- 本月整體總評
- 各店總評
- 各分類健康度
- 必查異常數
- 客訴異常數

第二層顯示：

- 最近巡店
- 待改善任務
- 低分排行
- 店別風險比較

---

## 9. 通知觸發條件定稿

通知分成三層：

- `高`
- `中`
- `低`

### 9.1 高優先級通知

觸發條件：

- 新增巡店後出現 `必查項目 = C`
- 新增巡店後出現 `客訴項目 = C`
- 新增巡店後整店總評為 `C`

### 9.2 中優先級通知

觸發條件：

- `本月加強` 題目持續落在 `B / C`
- 同題連續第 `2 週` 低分
- 同店待改善任務超過門檻

### 9.3 低優先級通知

觸發條件：

- 一般巡店完成通知
- 每週摘要
- 常規營運摘要

### 9.4 通知通道建議順序

建議開發順序：

1. 先做站內摘要 / 報表提示
2. 再做 Email 或外部通知
3. 最後才做 LINE

---

## 10. 頁面呈現規則

### 10.1 巡店歷史

每筆巡店至少顯示：

- 巡店日期
- 店別
- 巡店人
- 總評 `A / B / C`

### 10.2 巡店明細

至少顯示：

- 總評 `A / B / C`
- 分類等級
- 單題 `A / B / C`
- 總評調整原因
- 標籤 badge

### 10.3 報表

至少顯示：

- 各分類平均與等級
- 各店總評比較
- 必查 / 本月加強 / 客訴項目異常統計
- 低分排行
- 連續低分排行

---

## 11. 實作優先順序

固定順序如下：

1. A / B / C 全系統顯示一致化
2. 標籤在改善追蹤 / 報表 / 摘要頁完整產品化
3. QA 測試資料清理
4. 管理面與核心流程 E2E 補齊
5. 通知機制
6. 客訴資料庫串接
7. LINE / PDF / 其他延伸功能

---

## 12. 已拍板決策

這份 v1 目前固定如下：

1. `A / B / C` 是正式顯示語言
2. 必查 / 本月加強 / 客訴項目都會影響總評
3. 客訴項目預設看最近 `30 天`
4. 店長首頁第一眼要先看到：
   - 本店總評
   - 最弱分類
   - 待改善任務
   - 必查異常
   - 客訴異常
5. 通知先做站內與摘要邏輯，再考慮 LINE
